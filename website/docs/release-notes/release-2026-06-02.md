# Release 02/06/2026

## Release Notes

### New

- **🧠 Code Graph & 📚 LLM Wiki — the headline feature.** Agents no longer *explore* your codebase on every task — they already know it. A deterministic Tree-sitter map (`.code-review-graph/graph.db`, per-developer, not committed) plus an auto-generated, **committed** knowledge base (`docs/llm-wiki/`) ground every planner, implementer, and reviewer before they work. The result: faster delivery, lower token cost, and fewer hallucinations. → **See [Code Graph & LLM Wiki](../architecture/code-graph-and-wiki.md) for the full architecture.**
- **🔗 Multi-repo support.** The framework natively understands multi-repo workspaces — a single project made of several Git repositories side by side. One graph DB and one wiki at the workspace root index every child repo together. → **See the [Multi-Repository Setup](../workflows/initialize-project.md#multi-repository-setup) guide.**
- **New language skills.** Orientation skills for React Native, Expo, Scala, Swift, and Ruby (`mastering-react-native-skill`, `mastering-expo-skill`, `mastering-scala-skill`, `mastering-swift-skill`, `mastering-ruby-skill`) — routing work to the right upstream marketplace plugins while applying Qubika conventions — backed by new Kotlin, Scala, and Swift language configs.
- **🧱 Databricks / Lakehouse skill suite.** A full set of data-platform skills now ships under `skills/080-cloud-platforms/`, encoding Qubika's conventions for building on Databricks. It covers:
  - **Foundation & governance** — `qubika-databricks-sdk` (profile-based auth, async-safe `WorkspaceClient` patterns), `qubika-unity-catalog-governance` (catalog/schema/table naming, tagging, access control), and `qubika-uc-migration` (Hive Metastore → Unity Catalog via UCX).
  - **Pipelines & architecture** — `qubika-medallion-architecture` (Bronze/Silver/Gold on Delta Lake), `qubika-streaming-pipelines` (Auto Loader + Structured Streaming), and `qubika-kafka-patterns` (Schema Registry, offsets/DLQ/dedup, EventHub Kafka mode).
  - **Ingestion** — `qubika-federated-ingestion` (SQL Server/Oracle/Postgres/Redshift/Synapse via Lakefed), `qubika-saas-api-ingestion` (HubSpot/Salesforce/Zendesk/Stripe with secret-scope auth + watermarks), and `qubika-lakebridge` (transpile Snowflake/Oracle/Teradata/T-SQL/DataStage SQL to Databricks).
  - **AI on the Lakehouse** — `qubika-databricks-ai-functions` (SQL `ai_query`/`ai_classify`/`ai_extract`/`ai_summarize` for Gold-layer LLM enrichment) and `qubika-databricks-vector-search` (managed Vector Search indexes for RAG, with tenancy-isolated metadata filters).
  - **Quality, contracts & ops** — `qubika-data-quality` (DQX + Delta/DLT expectations), `qubika-data-contracts` (ODCS contracts via Ontos), `qubika-monitoring-observability` (job alerts, freshness, SLA tracking), and `qubika-jira-workflow` (Qubika DE board ticket-to-pipeline flow). These are auto-installed on `initialize-project` for projects detected as using Databricks.
- **Visual verification agent.** A new `visual-verifier` agent compares screenshots of UI changes against their expected designs and returns actionable fix suggestions when they drift.
- **New workflow skills.** `/fetch-ticket-context`, `/generate-test-cases`, `/apply-pr-feedback`, `/repo-fanout-pr` (multi-repo PR creation), and `/mastering-github-cli`.
- **Generated convention skills.** `code-conventions`, `testing-conventions`, and `multi-file-workflows` are now generated per-project so agents follow *your* conventions, not generic ones.
- **`/triage-incident` — SRE incident triage.** Paste an error, alert, or outage description and get a root-cause analysis grounded in Datadog signals, AWS data, Confluence history, and your codebase, plus a step-by-step runbook saved to markdown. All MCP integrations are optional — the skill degrades gracefully when any are absent.
- **Documentation website.** A full Docusaurus site now ships under `website/docs/` — getting-started, guides, reference, workflows, security, and architecture.

### Changed

- **`/implement-ticket` — graph-aware & multi-repo.** Runs the `ensure-context.sh` pre-flight, dispatches **multiple implementers in parallel**, is multi-repo aware, and auto-runs `/wiki-refresh` in Phase 8.5 so the shared wiki stays current after every ticket.
- **`pr-reviewer` & `security-review` — graph-backed and hardened.** Reviews pull grounded context from the graph. `security-review` is now a hybrid **SAST + LLM-triage** pipeline with dedicated agents for OWASP A01–A10 plus secrets and deserialization, emitting SARIF + JSON + a human report.
- **Jira-comment ingestion.** `/create-sdd-ticket` and `/implement-ticket` now read Jira comments for richer context.
- **Langfuse observability.** Metrics and tracing wiring is available for QAF flows.

### Fixed

- Service-detection accuracy — unreal/duplicate services are filtered out.

## Migration guide — a full rebuild is required

To use the new graph + wiki architecture you must **re-initialize** your project once. There is no incremental upgrade path: the Code Graph is built fresh from your source, and the LLM Wiki is generated from that graph. The graph is **per-developer and not committed**, so every machine builds its own.

> The short version: pull the latest framework, then run `initialize-project.sh`. The first run builds the graph and generates the wiki; from then on `ensure-context.sh` keeps them fresh automatically.

---

## What should I do? — pick your scenario

### Scenario 1 — "First time on this version (clean project)"

```bash
cd /path/to/your/project/qubika-agentic-framework
git pull
cd ..
./qubika-agentic-framework/scripts/initialize-project.sh
```

This builds the Code Graph, generates the LLM Wiki, and installs all the new skills. Run it from the directory that contains your repo.

### Scenario 2 — "I already have `.claude/` initialized" (recommended path)

Because the architecture changed substantially, the cleanest path is a **clean rebuild**: back up, delete, regenerate, then reconcile your custom edits.

```bash
# 1. Back up your current config (and any existing wiki)
cp -r .claude ".claude.bak-$(date +%Y%m%d)"
[ -d docs/llm-wiki ] && cp -r docs/llm-wiki docs/llm-wiki.bak

# 2. Delete the old config, transient artifacts, and any stale graph
rm -rf .claude .claude-temp .code-review-graph

# 3. Regenerate everything from scratch
./qubika-agentic-framework/scripts/initialize-project.sh

# 4. Reconcile — diff the backup against the freshly generated config
#    and re-apply only the custom skills/agents/CLAUDE.md edits you want to keep
diff -ruq ".claude.bak-$(date +%Y%m%d)" .claude
```

In step 4, copy back any **intentional** customizations (custom skills, edited agents, hand-written `CLAUDE.md` sections). Everything else is regenerated and current — leave it as-is. Once you're happy, delete the `.bak` folders.

### Scenario 3 — "I have a multi-repo workspace"

Run the initializer from the **workspace root** (the parent directory that contains all the repos), not inside a single repo. One graph DB and one wiki will index every child repo together. Full steps: [Multi-Repository Setup](../workflows/initialize-project.md#multi-repository-setup).

```bash
cd /path/to/workspace-root   # the folder holding repo-a/, repo-b/, ...
./qubika-agentic-framework/scripts/initialize-project.sh
```

### Scenario 4 — "I've already initialized — I just want fresh docs"

If you're on the new architecture and only want the wiki updated (not a full rebuild), run:

```bash
/wiki-refresh
```

It diffs each repo since its last-indexed commit and surgically edits only the pages whose facts drifted. The graph itself is kept fresh automatically by `ensure-context.sh` on every QAF flow.

## Using Codex (OpenAI) instead of Claude?

Every scenario above applies — just substitute the provider's paths and add the provider flag:

- `.claude/` → `.codex/`, `CLAUDE.md` → `AGENTS.md`, `.claude-temp/` → `.codex-temp/`
- Append `--provider codex` to every `initialize-project.sh` invocation.

So the clean-rebuild delete becomes `rm -rf .codex .codex-temp .code-review-graph` and the regenerate step becomes `./qubika-agentic-framework/scripts/initialize-project.sh --provider codex`.
