# Release 11/06/2026 — v3.0.1

> **Patch release.** This builds on **v3.0.0** (the Code Graph & LLM Wiki release). Unlike v3.0.0, **no full rebuild is required** — most of this ships through a plain `sync-framework-resources.sh`. See [What should I do?](#what-should-i-do--pick-your-scenario) below.

## Release Notes

### New

- **🔷 Azure DevOps support.** The framework is no longer GitHub-only. Two new skills ship: **`mastering-azure-devops-cli`** (Azure DevOps CLI mastery — work items, repos, pipelines) and **`create-pr-azure`**, which opens production-ready Azure DevOps Pull Requests with work-item linking, artifact collection, and reviewer assignment.
- **🧭 Auto-detected version-control host.** A new `software_version_control` field in the `framework-config.json` stack profile is populated automatically by reading your git `origin` remote and classifying the host (GitHub, GitLab, Azure DevOps, Bitbucket). `/implement-ticket` and `/repo-fanout-pr` now **route to the correct PR skill** (e.g. `create-pr` for GitHub vs `create-pr-azure` for Azure DevOps) based on the detected host — instead of assuming GitHub. Detection is repo-level and stack-agnostic; an unrecognized or missing remote simply leaves the field unset.
- **🧪 `qa-camunda` skill.** A new QA skill, triggered on `camunda` / `bpmn`, that walks an engineer through testing Camunda Wayfinder branch changes against a staging environment: starting process instances via the REST API and validating execution (activity tree, variables, jobs, incidents, history) before reporting results. The Camunda host is resolved at runtime via the `CAMUNDA_REST_BASE` environment variable, so the skill is safe to ship across every project.

### Changed

- **Richer, ground-truth `CLAUDE.md` generation.** Phase 3 synthesis and the tech-stack, directory-structure, and essential-commands renderers were substantially improved so generated `CLAUDE.md` files are denser and grounded in the real repository tree — fewer fabricated rows, more of the conventions that actually matter to your project.
- **`playwright-e2e-automation` skill — stronger assertions.** Added a dedicated "better assertions" section so generated E2E tests assert on meaningful, stable conditions rather than brittle ones.

### Fixed

- **Multi-repo artifact pollution.** Every `.claude-temp` / `.codex-temp` write is now anchored to the **workspace root** (resolved cwd-independently), and `--artifacts-dir` is threaded down into `/pr-reviewer` and `/security-review` — so ticket artifacts can no longer leak into a child repo and pollute its PR. Defense-in-depth `.git/info/exclude` entries keep transient temp dirs out of every child tree.
- **Portability check no longer false-fails on `settings.local.json`.** The Phase 6 validator was tripping on this per-developer, CLI-managed file; it is now carved out (never a hard failure) and instead surfaces an actionable `git rm --cached` warning if it happens to be git-tracked.
- **Deterministic, portable `framework-config.json`.** Phase-1 evidence is kept out of the committed config, embedded absolute paths are relativized, ordering is deterministic, and a stable `last_sync` marker is preserved — so the committed config no longer churns or produces spurious diffs across developer machines.
- **Removed the deprecated `MultiEdit` tool** from the implementer agent definitions (it is no longer part of the toolset).
- **Fixed the "Claude CLI version is required" startup error** that could block initialization.
- **Ruby project initialization** — added the missing `build` and `typecheck` commands so Ruby projects initialize cleanly.
- **`doc-updater` no longer halts `/implement-ticket`** — it delegates to a subagent instead of cutting off task execution.
- **`find_python` logging routes to stderr**, so it no longer pollutes stdout.

- **Unified non-portable-path regex.** The pattern that detects machine-specific absolute paths (`/Users/<name>/`, `/home/<name>/`) was duplicated across four layers — the runtime path resolver, the Phase 6 portability validator, the Zod refinement, and the synthesis-body validator. It now derives from a single source of truth (`portable-paths/patterns.ts`), so the matchers can no longer drift apart.

## This is a patch — no rebuild required

v3.0.0 required a one-time full re-initialization to adopt the Code Graph + LLM Wiki architecture. **v3.0.1 does not.** If you're already on v3.0.0, the new skills and the multi-repo artifact fix arrive through a normal sync; a re-init is only needed if you want the improved `CLAUDE.md` generation, the deterministic config, and version-control host auto-detection baked into your project.

## What should I do? — pick your scenario

### Scenario 1 — "I'm on v3.0.0 and just want the new skills" (recommended)

```bash
cd /path/to/your/project/qubika-agentic-framework
git pull
cd ..
./qubika-agentic-framework/scripts/sync-framework-resources.sh
```

You'll get the new **Azure DevOps** skills (`mastering-azure-devops-cli`, `create-pr-azure`), the **`qa-camunda`** skill, and the **multi-repo artifact-anchoring** fix in `/implement-ticket`, `/pr-reviewer`, and `/security-review`. Your `CLAUDE.md`, generated config, graph, and wiki are left untouched.

### Scenario 2 — "I want the improved CLAUDE.md, deterministic config, and VCS auto-detection"

These ship through `initialize-project` (Phase 1 detection + Phase 3 synthesis), so re-initialize to pick them up:

```bash
# back up your current .claude folder first if you've edited it
cp -r .claude ".claude.bak-$(date +%Y%m%d)"

./qubika-agentic-framework/scripts/initialize-project.sh
```

After this, `framework-config.json` gains the `software_version_control` field (so PR routing picks the right host automatically), the committed config stops churning across machines, and your `CLAUDE.md` is regenerated with the richer, grounded renderers. If you're happy with your current `CLAUDE.md`, **skip this** — Scenario 1 is enough.

### Scenario 3 — "I have a multi-repo workspace"

Run Scenario 1 to get the artifact-anchoring fix immediately — it stops ticket artifacts from leaking into child repos. To also adopt the deterministic config and improved `CLAUDE.md`, re-init from the **workspace root** (the parent directory that contains all the repos):

```bash
cd /path/to/workspace-root   # the folder holding repo-a/, repo-b/, ...
./qubika-agentic-framework/scripts/initialize-project.sh
```

## Using Codex (OpenAI) instead of Claude?

Every scenario above applies — just substitute the provider's paths and add the provider flag:

- `.claude/` → `.codex/`, `CLAUDE.md` → `AGENTS.md`, `.claude-temp/` → `.codex-temp/`
- Append `--provider codex` to every `initialize-project.sh` invocation.

The multi-repo artifact-anchoring fix applies identically to `.codex-temp/`.
