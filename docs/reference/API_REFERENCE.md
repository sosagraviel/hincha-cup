# API Reference

Complete reference for skills, agents, and utilities in the AI Agentic Framework.

> **Invocation**: user-invokable skills use `/skill-name` in Claude Code and `$skill-name` in Codex CLI. In Codex, run `/skills` to list the skills loaded in the current session. Skill definitions are identical between providers — only the prefix and the generated config directory (`.claude/` vs `.codex/`) differ.

## Skills

Skills are reusable knowledge modules that provide context and patterns to AI agents.

### Foundation Skills

**project-context**
- **Purpose**: Deep understanding of your specific project
- **Contains**: Architecture, patterns, conventions, testing setup
- **Generated by**: `/initialize-project` command
- **Used by**: All agents
- **Location**: `.claude/skills/project-context/SKILL.md`

**initialize-project**
- **Purpose**: Analyzes codebase and sets up framework
- **Actions**: Stack detection, skill copying, agent generation
- **Command**: `/initialize-project`
- **Output**: CLAUDE.md, project-context, agents

---

### Development Workflow Skills

**implement-ticket**
- **Purpose**: Full feature implementation workflow
- **Phases**: Analysis → Planning → Implementation → Testing → PR
- **Type**: User-invokable skill
- **Invocation**:
  - Claude Code: `/implement-ticket <TICKET_ID>`
  - Codex CLI: `$implement-ticket <TICKET_ID>`
- **Time**: 5-15 minutes per ticket
- **Used by**: All implementation agents

**fetch-ticket-context**
- **Purpose**: Retrieves ticket details from issue tracker
- **Supports**: Jira, GitHub Issues, Linear
- **Command**: `/fetch-ticket-context <TICKET_ID>`
- **Output**: Ticket context for planning

**code-quality-check**
- **Purpose**: Runs linting, type checking, tests, coverage
- **Command**: `/code-quality-check`
- **Checks**: Lint errors, type errors, test failures, coverage gaps
- **Time**: 1-3 minutes

**create-pr**
- **Purpose**: Creates GitHub pull request
- **Command**: `/create-pr`
- **Includes**: Code changes, test results, coverage report
- **Time**: 30 seconds

## Agents

Agents are AI assistants specialized for specific tasks.

### Planning Agents

**planner**
- **Purpose**: Creates implementation plans for medium-complexity tickets
- **Model**: Opus (highest reasoning capability)
- **Skills**: project-context, analyze-requirements, mastering-[all-detected-languages]
- **Context**: Architecture-aware across all project languages
- **Output**: Implementation plan with file changes and test strategy
- **When used**: Risk score 31-70 (medium complexity)

**architect**
- **Purpose**: Detailed architecture planning for high-risk tickets
- **Model**: Opus
- **Skills**: project-context, analyze-requirements, design-doc-mermaid, mastering-[all-detected-languages]
- **Context**: Full system architecture awareness
- **Output**: Architecture document, diagrams, implementation plan
- **When used**: Risk score 71+ (high complexity)

---

### Implementation Agents

**implementer-[language]**
- **Purpose**: Implements code in specific language following project conventions
- **Model**: Sonnet (fast, cost-effective)
- **Skills**: project-context, mastering-[language], [framework-skills]
- **Example**: `implementer-typescript` uses project-context, mastering-typescript, react-frontend
- **Output**: Production-ready code following YOUR patterns
- **Generated**: One per detected language (TypeScript, Python, Go, etc.)

**Multi-language routing**: Framework automatically routes files to correct implementer based on file extension.

---

### Testing Agents

**tester-unit-[language]**
- **Purpose**: Generates unit and integration tests
- **Model**: Sonnet
- **Skills**: project-context, mastering-[language], [test-framework-skills]
- **Example**: `tester-unit-typescript` uses project-context, mastering-typescript, jest-coverage-automation
- **Output**: Comprehensive tests with 80%+ coverage
- **Generated**: One per detected language

**tester-e2e-[language]**
- **Purpose**: Generates end-to-end tests
- **Model**: Sonnet
- **Skills**: project-context, mastering-[language], [e2e-framework-skills]
- **Example**: `tester-e2e-typescript` uses project-context, mastering-typescript, playwright-e2e-automation
- **Output**: E2E tests for critical user flows
- **Generated**: If E2E framework detected

---

### Review Agents

**security-reviewer-[language]**
- **Purpose**: Security analysis and vulnerability detection
- **Model**: Opus (deep analysis required)
- **Skills**: project-context, security-review, mastering-[primary-language]
- **Output**: Security report, vulnerability list, remediation steps
- **Generated**: For primary project language

## Commands

Invocations available through the Skill tool (e.g., `/implement-ticket`). All framework workflows are user-invokable skills — native slash commands are no longer shipped by the framework.

### Project Setup

**`/initialize-project`**
- **Purpose**: One-time project setup
- **Actions**:
  1. Detects tech stack (languages, frameworks, tools)
  2. Analyzes codebase structure and patterns
  3. Generates CLAUDE.md and project-context
  4. Copies stack-specific skills
  5. Creates custom agents
- **Time**: ~2 minutes
- **Run**: Once per project
- **Output**: `.claude/` directory with documentation and agents

---

### Feature Development

**`/implement-ticket <TICKET_ID>`** *(user-invokable skill)*
- **Purpose**: Full feature implementation
- **Phases**:
  1. Fetch ticket context
  2. Assess risk
  3. Create plan (if medium/high risk)
  4. Implement code
  5. Generate tests
  6. Run quality gates
  7. Create PR
- **Time**: 5-15 minutes
- **Arguments**:
  - `<TICKET_ID>`: Ticket key (e.g., PROJ-123)
  - `--resume`: Resume from checkpoint (optional)

**`/fetch-ticket-context <TICKET_ID>`**
- **Purpose**: Get ticket details before implementation
- **Fetches from**: Jira, GitHub, Linear (auto-detected)
- **Time**: ~10 seconds
- **Output**: Ticket context displayed in Claude Code

**`/create-pr`**
- **Purpose**: Create GitHub pull request
- **Requires**: Git changes committed
- **Includes**: Changes, test results, coverage
- **Time**: ~30 seconds
- **Output**: PR URL

---

### Quality Assurance

**`/code-quality-check`**
- **Purpose**: Run all quality checks
- **Checks**:
  - Linting (ESLint, Pylint, etc.)
  - Type checking (TypeScript, mypy, etc.)
  - Tests (Jest, Pytest, etc.)
  - Coverage (aim for 80%+)
- **Time**: 1-3 minutes
- **Output**: Pass/fail status, errors if any

---

### Utilities

**`/start-task <TICKET_ID>`**
- **Purpose**: Create isolated git worktree
- **Use case**: Work on multiple tickets in parallel
- **Output**: New worktree directory

**`/end-task <TICKET_ID>`**
- **Purpose**: Clean up worktree
- **Actions**: Removes worktree directory
- **Requires**: Changes committed or PR created

## Project Structure

Understanding the `.claude/` directory created by initialization.

```
.claude/
├── settings.json              # Framework configuration
├── CLAUDE.md                  # Quick reference guide (30-200 lines)
│
├── skills/                    # Stack-specific skills
│   ├── project-context/       # Your project's context
│   ├── mastering-typescript/  # TypeScript patterns
│   ├── react-frontend/        # React patterns
│   └── jest-coverage-automation/  # Jest testing
│
└── agents/                    # Custom AI agents
    ├── planner.md            # Medium-risk planning
    ├── implementer-typescript.md  # TypeScript implementation
    ├── tester-unit-typescript.md  # TypeScript testing
    └── security-reviewer-typescript.md  # Security review
```

### Generated Files

**CLAUDE.md** (project root)
- Quick reference for AI agents
- Contains: Stack summary, patterns, testing conventions
- Size: 30-200 lines (intentionally concise)
- Updated: Each time `/initialize-project` runs

**project-context/SKILL.md** (.claude/skills/)
- Deep context about your project
- Contains: Architecture, modules, data flows, conventions
- Size: 50-800 lines (comprehensive, scales with project complexity)
- Updated: Each time initialization runs

## Skill Selection Logic

How the framework decides which skills to copy:

### Language Detection

```
If tsconfig.json exists → Copy mastering-typescript
If pyproject.toml exists → Copy mastering-python-skill
If go.mod exists → Copy mastering-go
... (8+ languages supported)
```

### Framework Detection

```
If react in package.json → Copy react-frontend
If @nestjs/core in package.json → Copy nestjs-patterns
If django in requirements.txt → Copy django-patterns
... (40+ frameworks supported)
```

### Test Framework Detection

```
If jest in package.json → Copy jest-coverage-automation
If playwright in package.json → Copy playwright-e2e-automation
If pytest in pyproject.toml → Copy pytest-patterns
... (10+ test frameworks supported)
```

### Monorepo Handling

```
If pnpm-workspace.yaml exists → Detect workspaces
For each workspace:
  Detect languages → Copy language skills
  Detect frameworks → Copy framework skills
  Detect test frameworks → Copy testing skills
```

**Result**: Only skills relevant to YOUR stack are copied (10-20 skills instead of 50+)

## Agent Skill Mapping

Which skills each agent receives:

### Planner (Architecture-Aware)

```yaml
skills:
  - project-context          # Universal
  - analyze-requirements     # Planning core
  - design-doc-mermaid       # Planning core
  - mastering-typescript     # All detected languages
  - mastering-python-skill   # All detected languages
  - developing-with-docker   # Infrastructure
```

**Why**: Needs full architecture awareness across all project languages.

---

### Implementer-TypeScript (Language-Specific)

```yaml
skills:
  - project-context          # Universal
  - mastering-typescript     # THIS language only
  - react-frontend           # Detected frontend framework
  - atomic-design-react      # React patterns
```

**Why**: Focused on TypeScript implementation, ignores Python/Go/etc.

---

### Tester-Unit-TypeScript (Testing-Specific)

```yaml
skills:
  - project-context          # Universal
  - code-quality-check       # Quality core
  - mastering-typescript     # Language for tests
  - jest-coverage-automation # Detected test framework
```

**Why**: Focused on TypeScript testing with Jest.
